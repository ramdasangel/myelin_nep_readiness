## 1. Overview

The **Stage 5 — Micro-Intervention Task Mapping Dashboard** is a self-contained HTML file that visualises which tasks 169+ Deccan Education Society (DES) users selected at enrollment, their role and branch distribution, FP/depth coverage patterns, and the enrollment timeline. It must be regenerated whenever new users enroll or the `micro_intervention_report.csv` is refreshed.

## 3. Data Sources

### 3.1 Primary: `micro_intervention_report.csv`

Generated by `scripts/micro_intervention_report.js` running on prod.

| Column | Description |
|--------|-------------|
| `UserId` | MongoDB ObjectId |
| `FirstName`, `LastName`, `FullName` | User display name |
| `Role` | Teacher / Leader / blank |
| `SchoolName` | Filtered to "Deccan Education Society" |
| `BranchName`, `BranchCode` | School branch info |
| `TasksChosenCount` | Number of tasks selected |
| `TaskList` | Pipe-delimited: `T01-Notice One Learner \| T03-...` |
| `DatesLogged` | Pipe-delimited dates of daily logs |
| `TotalDaysLogged` | Count of distinct days logged |

**Location:** `output/micro_intervention_report.csv`

### 3.2 Static: `MAPPING_DATES` Dictionary

A Python dict in `scripts/build_task_mapping_dashboard.py` mapping `userId → ISO timestamp` (e.g., `'2026-02-03T10:12:11.847Z'`). Used to drive the **Enrollment Timeline** chart and display "Mapped On" date per user.

**Source:** `UserTaskMapping.createdAt` from MongoDB, extracted via `scripts/extract_mapping_dates_v2.js`.

**Current coverage:** 169 users across 5 enrollment batches (Feb 3–17, 2026).

### 3.3 Static: Task Metadata (hardcoded in builder)

12 tasks (T01–T12), each with FP, depth, and depth category. No changes expected unless the task library is updated.

## 4. Regeneration Steps

### Step 1: Refresh `micro_intervention_report.csv`

Always run this — it picks up new users and updated daily log counts.

```bash
# Upload extraction script to prod
scp -i ~/.ssh/myelin_pilot_key.pem \
  scripts/micro_intervention_report.js \
  ec2-user@13.200.74.24:~/myelin_stat_ro/

# Run extraction
ssh -i ~/.ssh/myelin_pilot_key.pem ec2-user@13.200.74.24 \
  "mongosh --port 27017 -u Shridhar -p 'ShriDhar@Myelin' \
   --authenticationDatabase pdea_pilot pdea_pilot \
   --quiet ~/myelin_stat_ro/micro_intervention_report.js \
   > ~/myelin_stat_ro/micro_intervention_report.csv"

# Download
scp -i ~/.ssh/myelin_pilot_key.pem \
  ec2-user@13.200.74.24:~/myelin_stat_ro/micro_intervention_report.csv \
  output/micro_intervention_report.csv

# Cleanup prod
ssh -i ~/.ssh/myelin_pilot_key.pem ec2-user@13.200.74.24 \
  "rm ~/myelin_stat_ro/micro_intervention_report.js \
      ~/myelin_stat_ro/micro_intervention_report.csv"
```

### Step 2: (Conditional) Update `MAPPING_DATES` for New Users

Only needed if **new users** have enrolled since the last build. To check:

```bash
python3 - << 'EOF'
import re, csv

html = open('output/task_mapping_dashboard.html').read()
existing_ids = set(re.findall(r"'([0-9a-f]{24})':\s*'20", html))

missing = []
with open('output/micro_intervention_report.csv', newline='', encoding='utf-8') as f:
    for row in csv.DictReader(f):
        if row['SchoolName'].strip() != 'Deccan Education Society':
            continue
        uid = row['UserId'].strip()
        if uid not in existing_ids:
            missing.append(uid)

print(f"New users needing mapping dates: {len(missing)}")
for uid in missing:
    print(f"  '{uid}'")
EOF
```

If the output shows `New users needing mapping dates: 0`, skip to Step 4.

**If there are new users:**

1. Add their IDs to `scripts/extract_mapping_dates_v2.js` (replace `newUserIds` list or append)
2. Run on prod:

```bash
scp -i ~/.ssh/myelin_pilot_key.pem \
  scripts/extract_mapping_dates_v2.js \
  ec2-user@13.200.74.24:~/myelin_stat_ro/

ssh -i ~/.ssh/myelin_pilot_key.pem ec2-user@13.200.74.24 \
  "mongosh --port 27017 -u Shridhar -p 'ShriDhar@Myelin' \
   --authenticationDatabase pdea_pilot pdea_pilot \
   --quiet ~/myelin_stat_ro/extract_mapping_dates_v2.js"

ssh -i ~/.ssh/myelin_pilot_key.pem ec2-user@13.200.74.24 \
  "rm ~/myelin_stat_ro/extract_mapping_dates_v2.js"
```

3. Append the output lines to the `MAPPING_DATES` dict in `scripts/build_task_mapping_dashboard.py` as a new batch, e.g.:

```python
# Batch 6 — YYYY-MM-DD (added YYYY-MM-DD)
'<userId>': '<ISO timestamp>',
```

### Step 3: (No manual date update needed)

Unlike `build_daily_progress_dashboard.py`, this builder uses `date.today()` — no hardcoded date to change.

### Step 4: Build the Dashboard

```bash
python3 scripts/build_task_mapping_dashboard.py
```

**Reads:** `output/micro_intervention_report.csv`

**Writes:** `output/task_mapping_dashboard.html`

**Expected output:**
```
Dashboard written to .../output/task_mapping_dashboard.html (100KB)
  169 users | Data through 2026-03-02
```

### Step 5: Verify

Open `output/task_mapping_dashboard.html` in a browser and confirm:
- Header shows the correct user count and date
- KPI row reflects updated total/active/teacher/leader counts
- Enrollment Timeline chart includes any new enrollment dates
- User table shows new users with correct task selections

## 6. Hardcoded Dependencies & Known Limitations

### 6.1 `MAPPING_DATES` Dictionary (lines 28–175 of builder)

Contains `userId → ISO timestamp` for all enrolled users. Must be updated manually when new users enroll (see Step 2 above).

**Fallback:** Users not in the dict will show `—` for "Mapped On" and will not appear in the Enrollment Timeline chart.

**Recommendation:** Extract mapping dates dynamically from `UserTaskMapping.createdAt` at build time, eliminating the need for this dict entirely.

### 6.2 School Filter

Line 179 of the builder: only processes rows where `SchoolName == 'Deccan Education Society'`. Other schools are silently excluded.

### 6.3 `TotalDaysLogged` from `micro_intervention_report.csv`

The dashboard's "Days Logged" column and "Users Who Logged ≥1 Day" KPI depend on `TotalDaysLogged` in the report CSV. This is refreshed in Step 1 — so the mapping dashboard picks up updated log counts without needing `daily_progress_full.csv`.
